# Requirements Document

## Introduction

This document covers four targeted improvement areas for the CulturePass Australia app: fixing authentication timing and error-message correctness; hardening AWS Amplify Gen 2 configuration to eliminate dual-source ambiguity and guest-read gaps; polishing cross-platform UI/UX for iOS, Android, and web; and improving the AI guidance documents (`Assistant.md`, `CLAUDE.md`) to reflect Cognito/Amplify Gen 2 patterns, test-runner setup, property-based testing, and clear definitions of done.

The app targets Expo SDK ~53 (React Native 0.79 + react-native-web), expo-router ~5, AWS Amplify Gen 2 (Cognito, AppSync/DynamoDB, S3, Lambda), TanStack Query 5, Zustand 5, Zod 3, NativeWind 4 + Tailwind 3, and TypeScript strict.

---

## Glossary

- **AuthProvider**: The React context component (`features/auth/AuthProvider.tsx`) that manages Cognito session state and broadcasts it via the `AuthContext`.
- **RequireAuth**: The route-guard component (`features/auth/RequireAuth.tsx`) that redirects unauthenticated users to `/sign-in`.
- **Hub event**: An Amplify `Hub.listen("auth", …)` callback event, specifically the `signedIn` payload emitted by `aws-amplify/utils` after a successful Cognito sign-in.
- **Cognito error code**: The `name` property of an error thrown by `aws-amplify/auth` functions (e.g. `NotAuthorizedException`, `UserNotConfirmedException`, `UsernameExistsException`, `CodeMismatchException`, `ExpiredCodeException`, `LimitExceededException`).
- **authMessage**: The utility function in `app/(auth)/sign-in.tsx` that maps a caught error to a human-readable string shown in the error banner.
- **configureAmplify**: The idempotent configuration function in `lib/aws/config.ts` that calls `Amplify.configure()` with Cognito and AppSync settings.
- **amplify_outputs.json**: The file generated by `npx ampx sandbox` or the Amplify CI backend phase that contains all runtime resource values for Cognito, AppSync, S3, and model introspection.
- **EXPO_PUBLIC_ vars**: Environment variables prefixed `EXPO_PUBLIC_` read at Metro bundle time by `lib/aws/config.ts`.
- **getDataAuthMode**: The function in `lib/aws/config.ts` that returns `"userPool"` or `"identityPool"` depending on the current session state.
- **AuthShell**: The shared scaffold component (`features/auth/AuthShell.tsx`) wrapping all authentication screens.
- **Screen**: The layout primitive (`components/ui/Screen.tsx`) that applies horizontal gutter and optional max-width constraint.
- **Design token**: A named value from `tailwind.config.js` or `lib/theme.ts` representing a color, spacing, radius, or typography scale entry.
- **Safe area**: The device region free of notches, Dynamic Island, home indicator, and status bar, provided by `react-native-safe-area-context`.
- **Touch target**: The tappable interactive region of a pressable element.
- **Skeleton state**: A placeholder UI rendered while asynchronous data is loading, consisting of animated shimmer blocks matching the shape of the expected content.
- **PBT**: Property-based testing — a test strategy that generates many random or structured inputs to verify invariants hold across the input space.
- **Definition of done**: The explicit, per-change-type checklist that must be satisfied before a change is considered complete.

## Requirements

### Requirement 1 — Auth Hub Event Timing Fix

**User Story:** As a user signing in to CulturePass, I want to be navigated to the home screen only after the app's auth state is fully updated, so that I am not immediately redirected back to the sign-in screen by RequireAuth.

#### Acceptance Criteria

1. WHEN the Amplify Hub emits a `signedIn` event, THE AuthProvider SHALL update `isAuthenticated` to `true` before `router.replace("/")` causes RequireAuth to evaluate session state.

2. WHEN `signIn.mutateAsync()` resolves successfully in `SignInScreen`, THE SignInScreen SHALL delay calling `router.replace("/")` until the AuthProvider's `isAuthenticated` flag is `true`, with a maximum wait of 3 000 ms before proceeding anyway.

3. WHEN the Amplify Hub emits a `signedIn` event, THE AuthProvider SHALL call `getAwsAuthUser()` and set `user` state before setting `initializing` to `false`.

4. WHEN RequireAuth evaluates and `initializing` is `true`, THE RequireAuth component SHALL render the loading placeholder and SHALL NOT emit a `<Redirect>` to `/sign-in`.

5. WHEN RequireAuth evaluates and `initializing` is `false` and `isAuthenticated` is `false`, THE RequireAuth component SHALL emit a `<Redirect href="/sign-in" />`.

6. WHEN RequireAuth evaluates and `initializing` is `false` and `isAuthenticated` is `true`, THE RequireAuth component SHALL render its `children` prop without any redirect.

### Requirement 2 — Cognito Error Code Mapping in authMessage

**User Story:** As a user who encounters a sign-in or sign-up error, I want to see a clear, friendly message that corresponds to the actual Cognito error, so that I understand what went wrong and how to fix it without seeing raw AWS exception names.

#### Acceptance Criteria

1. WHEN `authMessage` receives an error whose `name` property is `"NotAuthorizedException"`, THE authMessage function SHALL return the string `"That email or password isn't right."`.

2. WHEN `authMessage` receives an error whose `name` property is `"UserNotConfirmedException"`, THE authMessage function SHALL return the string `"Please confirm your email first — check your inbox."`.

3. WHEN `authMessage` receives an error whose `name` property is `"UsernameExistsException"`, THE authMessage function SHALL return the string `"An account with that email already exists."`.

4. WHEN `authMessage` receives an error whose `name` property is `"CodeMismatchException"`, THE authMessage function SHALL return the string `"That code isn't right — please check and try again."`.

5. WHEN `authMessage` receives an error whose `name` property is `"ExpiredCodeException"`, THE authMessage function SHALL return the string `"That code has expired — please request a new one."`.

6. WHEN `authMessage` receives an error whose `name` property is `"LimitExceededException"`, THE authMessage function SHALL return the string `"Too many attempts — please wait a few minutes and try again."`.

7. WHEN `authMessage` receives an error whose `name` property is `"InvalidPasswordException"`, THE authMessage function SHALL return the string `"Password does not meet the requirements."`.

8. WHEN `authMessage` receives an error that does not match any mapped Cognito error code, THE authMessage function SHALL return the `message` property of the error if the error is an `Error` instance, otherwise SHALL return `"Something went wrong."`.

9. THE authMessage function SHALL match error codes using the `name` property of the error object, not by substring-matching the `message` property.

### Requirement 3 — Session Persistence on Reload

**User Story:** As a returning user who reloads the app, I want to remain signed in without having to re-enter my credentials, so that the experience is seamless across sessions.

#### Acceptance Criteria

1. WHEN the AuthProvider mounts, THE AuthProvider SHALL call `getAwsAuthUser()` once and set `user` state from the result before setting `initializing` to `false`.

2. WHEN `getAwsAuthUser()` returns a non-null `AwsAuthUser` on mount, THE AuthProvider SHALL call `setDataSignedIn(true)` before `initializing` transitions to `false`.

3. WHEN `getAwsAuthUser()` returns `null` on mount (no persisted session), THE AuthProvider SHALL call `setDataSignedIn(false)` and set `initializing` to `false`.

4. WHILE `initializing` is `true`, THE AuthProvider SHALL continue rendering children so the root layout does not unmount active screens.

5. IF `getAwsAuthUser()` throws an unhandled exception during mount, THEN THE AuthProvider SHALL treat the result as `null` (signed out), set `initializing` to `false`, and log the error to the console without crashing the app.

### Requirement 4 — AWS Config Dual-Source Fallback

**User Story:** As a developer running a build where `.env` is missing or stale, I want `configureAmplify()` to fall back to `amplify_outputs.json` for all Cognito and AppSync fields, so that the app remains functional without manual env setup.

#### Acceptance Criteria

1. WHEN `configureAmplify()` is called and `EXPO_PUBLIC_COGNITO_USER_POOL_ID` is an empty string, THE configureAmplify function SHALL read `auth.user_pool_id` from `amplify_outputs.json` and use that value.

2. WHEN `configureAmplify()` is called and `EXPO_PUBLIC_COGNITO_APP_CLIENT_ID` is an empty string, THE configureAmplify function SHALL read `auth.user_pool_client_id` from `amplify_outputs.json` and use that value.

3. WHEN `configureAmplify()` is called and `EXPO_PUBLIC_COGNITO_IDENTITY_POOL_ID` is an empty string, THE configureAmplify function SHALL read `auth.identity_pool_id` from `amplify_outputs.json` and use that value.

4. WHEN `configureAmplify()` is called and `EXPO_PUBLIC_APPSYNC_ENDPOINT` is an empty string, THE configureAmplify function SHALL read `data.url` from `amplify_outputs.json` and use that value.

5. WHEN `configureAmplify()` is called and `EXPO_PUBLIC_AWS_REGION` is an empty string, THE configureAmplify function SHALL read `auth.aws_region` from `amplify_outputs.json` and use that value.

6. WHEN `configureAmplify()` is called and all `EXPO_PUBLIC_*` vars are present and non-empty, THE configureAmplify function SHALL use the `EXPO_PUBLIC_*` values and SHALL NOT read from `amplify_outputs.json` for those fields.

7. IF `amplify_outputs.json` is absent or malformed and all `EXPO_PUBLIC_*` vars are empty, THEN THE configureAmplify function SHALL return `false` without calling `Amplify.configure()` and SHALL log a descriptive error to the console.

### Requirement 5 — Hot-Reload Reconfiguration

**User Story:** As a developer iterating in an Expo dev server, I want `configureAmplify()` to reconfigure Amplify when the module reloads during hot-reload, so that stale config does not silently survive across development sessions.

#### Acceptance Criteria

1. WHEN the Metro bundler performs a full module reload (fast-refresh boundary crosses `lib/aws/config.ts`), THE configureAmplify function SHALL reset the `configured` flag and call `Amplify.configure()` again with the current resolved values.

2. WHEN `configureAmplify()` is called a second time with identical resolved config values, THE configureAmplify function SHALL no-op and return `true` without calling `Amplify.configure()` again.

3. WHEN `configureAmplify()` is called while `__DEV__` is `true` and the previously configured values differ from the current resolved values, THE configureAmplify function SHALL call `Amplify.configure()` with the new values and update the cached config fingerprint.

### Requirement 6 — AppSync Guest Identity Pool Auth Mode

**User Story:** As a guest (unauthenticated) visitor to CulturePass on the web or mobile, I want public lists (Events, Hubs, States, Councils) to load successfully, so that I can browse without creating an account.

#### Acceptance Criteria

1. WHEN `getDataAuthMode()` is called and `dataSignedIn` is `false`, THE getDataAuthMode function SHALL return `"identityPool"`.

2. WHEN `getDataAuthMode()` is called and `dataSignedIn` is `true`, THE getDataAuthMode function SHALL return `"userPool"`.

3. WHEN a guest AppSync request is made with `authMode: "identityPool"` and `identityPoolId` was not provided to `Amplify.configure()`, THE configureAmplify function SHALL have included `identityPoolId` and `allowGuestAccess: true` in the Cognito config block, so that the unauthenticated IAM role credentials are available.

4. WHEN `EXPO_PUBLIC_COGNITO_IDENTITY_POOL_ID` is empty and `amplify_outputs.json` provides `auth.identity_pool_id`, THE configureAmplify function SHALL configure `allowGuestAccess: true` using the fallback identity pool id, preserving the guest read capability.

5. IF `identityPoolId` is unavailable from both `EXPO_PUBLIC_*` vars and `amplify_outputs.json`, THEN THE configureAmplify function SHALL configure Amplify without `allowGuestAccess` and SHALL log a warning that guest reads will fail with "No federated jwt".

### Requirement 7 — Keyboard-Avoiding Behaviour on Auth Forms

**User Story:** As a user on iOS or Android typing into a sign-in or sign-up form, I want the keyboard to push the form content up rather than covering the input fields, so that I can see what I am typing and access all form controls.

#### Acceptance Criteria

1. WHEN the software keyboard appears on iOS while an auth form input is focused, THE AuthShell component SHALL render a `KeyboardAvoidingView` with `behavior="padding"` so that the card containing the inputs is not obscured.

2. WHEN the software keyboard appears on Android while an auth form input is focused, THE AuthShell component SHALL render a `KeyboardAvoidingView` with `behavior="height"` so that the card containing the inputs is not obscured.

3. WHEN the software keyboard is dismissed, THE AuthShell component SHALL restore the layout to its original scroll position without animation artifacts.

4. WHERE the platform is web, THE AuthShell component SHALL render without a `KeyboardAvoidingView` wrapper, as the browser handles keyboard layout natively.

---

### Requirement 8 — Safe Area Handling on Auth and Shell Screens

**User Story:** As a user on an iPhone with a notch or Dynamic Island, I want content to not be clipped by the device's safe area insets, so that all text, buttons, and navigation elements are fully visible.

#### Acceptance Criteria

1. THE Screen component SHALL apply `SafeAreaView` insets (top, bottom, left, right) to its outermost container so that content respects the device safe area on all supported iOS and Android models.

2. WHEN the app is rendered on a device with a Dynamic Island, THE TopBar component SHALL position its content below the Dynamic Island safe area top inset.

3. WHEN the app is rendered on a device with a home indicator, THE BottomTabBar component SHALL add bottom padding equal to the safe area bottom inset so that tab items are not obscured.

4. WHERE the platform is web, THE Screen component SHALL apply no safe area insets and SHALL use CSS layout defaults.

---

### Requirement 9 — Web Breakpoints on AuthShell and Cards

**User Story:** As a user accessing CulturePass on a wide-screen web browser, I want the authentication and card layouts to be appropriately constrained and centred, so that the UI does not stretch uncomfortably across large displays.

#### Acceptance Criteria

1. WHERE the platform is web and the viewport width is 640 px or wider, THE AuthShell component SHALL constrain its content column to a maximum width of 480 px and centre it horizontally.

2. WHERE the platform is web and the viewport width is 1 024 px or wider, THE Card component SHALL constrain its maximum width to 600 px when rendered within an AuthShell.

3. WHERE the platform is web, THE AuthShell component SHALL apply horizontal padding of at least 16 px on viewports narrower than 640 px to prevent content touching the screen edge.

4. WHERE the platform is web, THE Screen component SHALL support a `maxWidth` prop accepting `"form"` (480 px), `"content"` (720 px), and `"wide"` (1 200 px) named breakpoints applied as `max-w-*` Tailwind classes.

---

### Requirement 10 — Minimum Touch Target Size

**User Story:** As a user on a mobile device with a touchscreen, I want all interactive elements to be large enough to tap accurately, so that I do not accidentally activate the wrong control.

#### Acceptance Criteria

1. THE Button component SHALL have a minimum height of 44 pt on iOS and 48 dp on Android, conforming to platform human interface guidelines.

2. THE Pressable elements rendered in link footers (e.g. "Forgot your password?", "New here?") SHALL have a `hitSlop` value that expands the touch area to at least 44 × 44 pt on iOS and 48 × 48 dp on Android.

3. THE BottomTabBar tab items SHALL have a minimum touch target height of 44 pt inclusive of any padding.

4. WHERE the platform is web, THE Button component SHALL have a minimum height of 40 px and apply `cursor: pointer` to match web interaction conventions.

---

### Requirement 11 — Loading and Skeleton States

**User Story:** As a user waiting for content to load, I want to see a placeholder that matches the shape of the incoming content, so that the layout does not jump and I understand that data is loading.

#### Acceptance Criteria

1. WHEN a list of Hubs or Events is being fetched and no cached data exists, THE hub and event list components SHALL render Skeleton placeholder items in place of real cards for the duration of the fetch.

2. WHEN a profile is being fetched and no cached data exists, THE profile screen SHALL render Skeleton placeholder blocks in place of the avatar, name, and bio for the duration of the fetch.

3. WHEN `RequireAuth` renders and `initializing` is `true`, THE RequireAuth component SHALL render a full-screen Skeleton placeholder rather than a bare text "Loading…" string.

4. WHEN a mutation (sign-in, sign-up, hub create, event publish) is in the `isPending` state, THE submit Button component SHALL display a loading spinner and SHALL disable the button to prevent duplicate submissions.

5. THE Skeleton component SHALL use an animated shimmer effect using `react-native-reanimated` that cycles at 1 200 ms per cycle and uses the `linen` design token as the base color and `sand` as the highlight color.

---

### Requirement 12 — Error Feedback UI

**User Story:** As a user who encounters an error in a form or data fetch, I want a clear, visible error message that describes what went wrong, so that I know whether to retry or take a different action.

#### Acceptance Criteria

1. WHEN a mutation error occurs in any auth form, THE auth form screen SHALL display the error message in the AuthShell `error` prop banner within the same render cycle as the mutation's `onError` callback.

2. WHEN a data fetch error occurs in a list or detail screen, THE affected screen SHALL render an inline error state with a retry button that re-triggers the query.

3. WHEN an inline error state is displayed, THE error message SHALL not be the raw JavaScript `Error.message` string; THE screen SHALL map network and AppSync errors to one of: `"Couldn't load content — check your connection."`, `"Something went wrong. Tap to retry."`, or a more specific message if available.

4. IF a form submission error banner is visible and the user begins editing any input field, THEN THE error banner SHALL be cleared.

5. THE AuthShell error banner SHALL be visually distinct from the notice banner, using the `terracotta-50` background and `danger` border color versus `eucalyptus-50` and `eucalyptus-100` for notices.

---

### Requirement 13 — Consistent Design Token Usage

**User Story:** As a developer maintaining the CulturePass codebase, I want all components to reference named design tokens rather than hardcoded color or spacing values, so that visual changes can be made from a single source of truth.

#### Acceptance Criteria

1. THE components in `components/ui/` SHALL reference color values exclusively via Tailwind class names defined in `tailwind.config.js` or via the `colors` object from `lib/theme.ts`, and SHALL NOT contain hardcoded hex, `rgb()`, or numeric color literals.

2. THE components in `components/ui/` SHALL reference spacing and border-radius values exclusively via Tailwind utility classes or the `spacing` and `radius` tokens in `tailwind.config.js`, and SHALL NOT contain hardcoded numeric pixel values outside of `StyleSheet.create` blocks that mirror token values.

3. WHEN a new component is added to `components/ui/`, THE component SHALL use the `cn()` utility from `lib/utils/cn.ts` for conditional class composition and SHALL NOT use string concatenation to build class lists.

4. THE `country.*` color tokens in `tailwind.config.js` SHALL be used exclusively within `components/cultural/` components and SHALL NOT appear in `components/ui/` or screen files.

### Requirement 14 — Assistant.md and CLAUDE.md Structure and Amplify Gen 2 Coverage

**User Story:** As an AI assistant or new contributor reading the guidance documents, I want a clearly structured document with explicit Amplify Gen 2 gotcha coverage, so that I can navigate the codebase correctly without repeating known mistakes.

#### Acceptance Criteria

1. THE Assistant.md file SHALL contain a dedicated "Amplify Gen 2 / Cognito gotchas" section that documents: the `signedIn` Hub event timing race, `authMessage` Cognito code mapping, the dual-source config ambiguity between `EXPO_PUBLIC_*` vars and `amplify_outputs.json`, and the guest identity pool fallback gap.

2. THE CLAUDE.md file SHALL reference the "Amplify Gen 2 / Cognito gotchas" section of `Assistant.md` in its high-value gotchas list.

3. THE Assistant.md file SHALL contain a dedicated "Testing" section that documents the configured Vitest setup for unit and integration tests — the jsdom environment, the `react-native` → `react-native-web` alias (and the `Module._resolveFilename` patch that makes it apply to externalized CJS deps), the use of `react-test-renderer` for component tests, and the recommended configuration for running tests in CI.

4. THE Assistant.md file SHALL contain a dedicated "Property-based testing" subsection within the "Testing" section that documents: what PBT is, when to apply it (pure functions, validation schemas, data mappers, auth message mapping), the recommended library (`fast-check`), and at least two concrete example patterns applicable to this codebase.

5. THE CLAUDE.md file SHALL contain a "Definition of done" section that lists per-change-type checklists covering: schema changes, auth changes, UI-only changes, data-fetch changes, and documentation-only changes.

6. THE Assistant.md "Definition of done" section SHALL be updated to include: "No Cognito error code appears raw in the UI", "Keyboard-avoiding and safe area handling verified on iOS simulator and Android emulator", "All new interactive elements meet 44 pt / 48 dp minimum touch target", and "Skeleton or loading state added for any new async data fetch".

7. THE CLAUDE.md file SHALL document the Amplify Gen 2 deployment workflow patterns including: the relationship between `npx ampx sandbox`, `amplify_outputs.json`, and `.env`; when to use `EXPO_PUBLIC_*` vars vs relying on the outputs file; and how to verify that `configureAmplify()` is using the correct source.

8. THE Assistant.md file SHALL document all Cognito error codes handled by `authMessage`, their human-readable mappings, and a note that matching MUST use `error.name` not `error.message`.
